Expert Audit Synthesis

What to build into v2, what to defer, and what to skip

Architecture Adjustments (all experts agree)

Incorporate into the spec
Request queuing hx-sync="this:queue all" on the coding workspace — mandatory for rapid keypresses (3, 5, 7 in quick succession). Without this, annotations get lost.
idiomorph for text panel hx-swap="morph:innerHTML" on #text-panel only. Preserves scroll position, ~3ms cost. Add id="ann-{uuid}" to annotation spans for better diffing.
SortableJS lifecycle destroy() in htmx:beforeSwap, Sortable.create() in htmx:afterSwap. Prevents memory leak from orphaned document listeners.
CSS classes for annotations Pre-generate .ace-code-{id} classes instead of inline rgba() styles. 30% faster paint, smaller HTML fragments.
OOB swaps pattern Single response returns primary target + OOB updates. E.g., POST /annotate returns text panel + annotation list in one response. No race conditions.
Keyboard → hidden trigger Shortcut keys trigger hidden HTMX elements (not htmx.ajax()) to preserve request queuing and swap behaviour.
UTF-16 conversion in JS Move offset conversion to bridge.js. Browser creates UTF-16, convert to codepoints before POST. Eliminates offset.py.

UX Features for v2

Build into initial release (high impact, low effort)
Quick Coding Press Q to apply last-used code to current selection. Single highest-impact feature missing from ACE. All major competitors have this.
Code search/filter Dual-purpose input: type to filter existing codes, Enter to create new. Critical once codebook exceeds ~30 codes. Like Delve's approach.
Shortcut cheat sheet Press ? to show overlay of all shortcuts. Standard in every keyboard-heavy app.
Code apply flash Brief background pulse on the code row when applied via keyboard. Confirms which code was applied — catches mistyped shortcuts.
Annotation count in delete "This will remove 47 annotations across 12 sources" instead of vague warning. One query.
F for flag toggle Consistent with G for grid. Currently mouse-only.
Code frequency counts Small count badge per code showing usage in current source. At-a-glance density feedback.

Text Panel (reading surface)

Optimised for extended reading of source documents
.ace-text-content { font-size: 15px; line-height: 1.75; max-width: 72ch; /* optimal reading measure */ padding: 24px 32px; /* generous — 2x sidebar padding */ -webkit-font-smoothing: antialiased; } .ace-text-content::selection { background: rgba(0, 0, 0, 0.12); /* subtle, brand-consistent */ } Annotation opacity: 0.22 (down from 0.30) Reduces visual noise with many overlapping codes.

Deferred to Later Versions

Good ideas, but not v2 scope
Coding stripe gutter Vertical coloured bars in margin (like MAXQDA). High impact but high effort — needs pixel-position calculation from character offsets. v3 feature.
Sticky code mode "Code first, then select" for focused coding. Good for deductive workflows. Can layer in after Quick Coding is proven.
Memo field UI Inline memo editor for annotations. DB column exists. UI needs design thought.
In Vivo coding Create code from selected text. Nice for open coding. Small feature, defer to keep scope tight.
Dark mode Architecture supports it (custom properties). Implementation deferred. One @media block when ready.
Source search/filter Search by display_id, filter by status. Good for 100+ source projects. Not critical for v2.
Persistent undo across nav Currently undo stack clears on source change. Could persist to server session. Adds complexity.
Code descriptions Optional description field in codebook. Schema change needed. Defer.

Performance Summary

HTMX on localhost is fast. Very fast.
Annotation roundtrip (POST → swap) 3-8ms vs ~20-60ms with NiceGUI WebSocket
idiomorph text panel diff 2-4ms 800 DOM nodes, preserves scroll
3x OOB swaps (text + list + header) <4ms Browser batches into one paint
Initial page load (JS + CSS) ~36 KB vs ~300-400 KB with NiceGUI/Vue/Quasar
Time to interactive 50-100ms vs ~200-500ms (no hydration, no WebSocket)

Ready for Spec?

The architecture is audited. The experts found no deal-breakers — only improvements to incorporate. The main risk (request queuing for rapid keypresses) has a clean solution (hx-sync).
