ACE v2 — Proposed Architecture

FastAPI + Jinja2 + HTMX + Vanilla JS. No build step. No abstraction layer.

Stack

Technology Choices
ServerFastAPI — ASGI, async, lightweight
TemplatesJinja2 — server-rendered HTML, template inheritance
InteractivityHTMX 2.x (~14 KB) — partial page updates via HTML fragments
Drag & dropSortableJS (~13 KB) — carried over from current app
Custom JSbridge.js — text selection, keyboard shortcuts, annotation flash
CSSPlain CSS — no framework, no utility classes, you write it directly
DatabaseSQLite — same .ace format, leaner schema
DistributionPython package (pip/uv) — desktop-packaging-ready later

How HTMX Replaces NiceGUI

Data Flow: Applying an Annotation

Current (NiceGUI):

JS mouseup → NiceGUI WebSocket event → Python handler → DB insert → NiceGUI re-renders Vue component → WebSocket pushes DOM update

New (HTMX):

JS mouseup → fetch POST /api/annotations (HTMX) → Python handler → DB insert → return HTML fragment → HTMX swaps into DOM

Same number of steps, but every step is inspectable: you can see the POST in Network tab, read the HTML response, and the swap is a simple innerHTML replacement. No WebSocket, no Vue, no virtual DOM.

What Stays, What Changes, What's New

Keeps (reused)
  • All models/ (annotation, codebook, coder, source, etc.)
  • All services/ (undo, palette, ICR, agreement, import/export)
  • db/ schema + migrations
  • bridge.js text selection logic
  • SortableJS drag-and-drop
  • .ace SQLite file format
  • Keyboard shortcut mappings
Changes (rewritten)
  • All page files → Jinja2 templates
  • NiceGUI event handlers → FastAPI route handlers
  • bridge.js event emission → HTMX requests
  • CSS (new design from scratch)
  • App startup (FastAPI instead of NiceGUI)
  • Storage (app.storage → server-side session or cookies)
New
  • routes/ — FastAPI routers for pages + HTMX API endpoints
  • schemas/ — Pydantic models for request/response validation
  • templates/ — Jinja2 templates with HTMX attributes
  • fragments/ — partial HTML responses for HTMX swaps

Pages — Build Order

Migration Sequence (simplest → hardest)
1 Landing page Simplest — buttons, dialogs, recent files list. Establishes base template + routing.
2 Import page File upload, preview table, column selection. Tests HTMX form handling.
3 Coding page The hard one — two-pane layout, text selection, annotations, DnD, shortcuts, undo/redo, source grid.
4 Agreement page File picker, validation, metrics dashboard, heatmap, export. Mostly display.

Key Design Decisions

  • No JavaScript build step — vendor HTMX + SortableJS as static files
  • Server-rendered HTML — Jinja2 templates, HTMX fragments for dynamic updates
  • Plain CSS — one file, you control every pixel, new visual design
  • Models/services untouched — proven logic, just add a thin API layer on top
  • Desktop-ready — no hardcoded URLs, clean startup/shutdown, structurally wrappable
